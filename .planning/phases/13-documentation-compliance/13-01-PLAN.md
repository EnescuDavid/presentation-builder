Create `docs/presentation-checklist.md` with the following structure (per D-03, D-04):

**Header:**
- Title: "Presentation Accessibility Checklist"
- Instructions: "Copy this template for each presentation. Check each item before distributing. Items marked [Auto] can be verified with the listed tool command. Items marked [Manual] require human review."
- Fill-in fields: Presentation name, Date, Checked by, Theme used

**Checklist sections grouped by content author task category (per D-03):**

1. **Images & Visual Content** (maps to 1.1.1, 1.4.5)
   - [ ] [Manual] Every `<img>` has descriptive `alt` text (not filename). Decorative images use `alt=""`.
     - Pass: All images have meaningful alt text or are marked decorative
     - Fail: Any image has `alt="image1.png"`, `alt="logo"`, or missing alt attribute
   - [ ] [Manual] Every `<canvas>` chart has an `aria-label` describing the data trend and key takeaway (e.g., `aria-label="Umsatzentwicklung 2023-2025: +23% Wachstum"`).
     - Pass: aria-label describes what the data shows
     - Fail: aria-label is generic like "Balkendiagramm" or missing
   - [ ] [Manual] Mermaid diagram containers have `aria-label` describing the process/relationship shown.
   - [ ] [Manual] Harvey ball ratings have text equivalents (not relying on visual dot fill alone).
   - [ ] [Manual] No essential text is embedded in images (use real HTML text instead).

2. **Color & Contrast** (maps to 1.4.1, 1.4.3, 1.4.11)
   - [ ] [Auto] Run `node tools/check-contrast.js --theme {theme-name}` -- all color pairs pass 4.5:1 (normal) and 3:1 (large text).
   - [ ] [Manual] Information is not conveyed by color alone -- charts use labels/patterns/shapes in addition to color.
   - [ ] [Manual] Custom colors (if any added beyond theme tokens) meet contrast ratios.

3. **Text & Structure** (maps to 1.3.1, 1.3.2, 2.4.6, 3.1.1, 3.1.2)
   - [ ] [Auto] Verify `lang` attribute on `<html>` matches presentation language (check: `grep 'lang=' presentation.html | head -1`).
   - [ ] [Manual] Mixed-language sections have appropriate `lang` attributes (e.g., `<span lang="en">stakeholder</span>` in German text).
   - [ ] [Manual] Heading hierarchy is correct: h1 only on title slide, h2 on content slides. No skipped levels.
   - [ ] [Manual] Reading order (DOM order) matches intended visual order on each slide.

4. **Navigation & Interaction** (maps to 2.1.1, 2.1.2, 2.4.3, 2.4.7)
   - [ ] [Manual] All slides are reachable via keyboard (arrow keys, space).
   - [ ] [Manual] No keyboard traps -- pressing Tab or arrow keys always moves forward.
   - [ ] [Manual] Focus indicators are visible on interactive elements (links, buttons).
   - [ ] [Manual] If auto-advance is enabled (`autoSlide`), verify it can be paused.

5. **Animations & Motion** (maps to 2.2.2, 2.3.1)
   - [ ] [Manual] No element flashes more than 3 times per second.
   - [ ] [Manual] Animations are decorative, not required to understand content.
   - [ ] [Manual] Presentation respects `prefers-reduced-motion` (test: enable reduced motion in OS settings, verify animations are suppressed).

6. **Accessible Export** (maps to 1.4.10, Section 10 non-web)
   - [ ] [Auto] Run `node tools/export-accessible.js {presentation.html}` -- produces linear HTML without reveal.js framework.
   - [ ] [Manual] Accessible export has meaningful heading structure (h1/h2/h3 hierarchy).
   - [ ] [Manual] Charts in accessible export have text descriptions replacing canvas elements.

7. **Title Quality** (maps to consulting best practice + 2.4.2)
   - [ ] [Auto] Run `node tools/read-titles.js {presentation.html}` -- verify all slides have action titles (complete sentences, not topic labels).
   - [ ] [Manual] HTML `<title>` element contains the presentation name.

**Footer:**
