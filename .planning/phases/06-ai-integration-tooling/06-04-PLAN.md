    XML body sections:

    - `<execution_flow>`:
      - Step 1: Read `brief.md` from the project folder. Identify the topic, audience type, and key messages.
      - Step 2: Determine mode. If brief mentions "research", "find data", "statistics", "benchmarks", or "real numbers" -> web research mode. Otherwise -> content structuring mode.
      - Step 3 (content structuring): Organize the user's bullet points into slide-sized chunks. Each chunk should have a clear headline, 3-5 supporting points, and a takeaway. Write to `research.md`.
      - Step 3 (web research): Use WebSearch to find relevant statistics, market data, quotes, and benchmarks. Verify sources. Structure findings as slide-ready content blocks. Write to `research.md`.
      - Step 4: Format `research.md` with one H2 per content block. Each block: headline, key data points, source attribution (if researched), suggested visualization (metric, chart, quote).
    - `<success_criteria>`: research.md exists, has one section per key message from brief, each section has structured content ready for the strategist.

    ```yaml
    ---
    name: presentation-builder
    description: Reads deck-plan.md and generates a complete reveal.js presentation HTML file using the framework's component templates and theme system
    tools: Read, Write, Edit, Bash, Glob, Grep
    model: sonnet
    ---
    ```

    XML body sections:
    - `<role>`: You are a presentation builder. You read a deck-plan.md and generate a complete, self-contained reveal.js presentation HTML file. You use the framework's component templates as reference patterns (not rigid molds -- improvise from them per Phase 2 decision), apply the specified theme, and ensure consulting-grade visual quality.
    - `<required_reading>`: Load these before building:
      1. `.claude/skills/build-presentation/references/component-catalog.md` (component patterns)
      2. `.claude/skills/build-presentation/references/theme-system.md` (token system)
      3. `.claude/skills/build-presentation/references/animation-guide.md` (animation classes)
      4. `.claude/skills/build-presentation/references/design-principles.md` (quality standards)
      5. `templates/_skeleton.html` (reveal.js config, master layer, output structure)
    - `<execution_flow>`:
      - Step 1: Read deck-plan.md from the project folder. Note: audience type, theme, total slides, per-slide component choices.
      - Step 2: Read `templates/_skeleton.html` for the base HTML structure (reveal.js config, master layer, CSS includes).
      - Step 3: For each slide in the plan, read the corresponding template from `templates/{component}.html`. Use the template as a reference pattern -- adapt the HTML structure to fit the planned content. Do NOT copy templates verbatim if the content requires layout adjustments.
      - Step 4: Read the theme file from `themes/{theme}/theme.css`. Apply theme CSS link in the HTML head.
      - Step 5: Assemble the full presentation.html. Structure: DOCTYPE -> head (meta, theme CSS, tokens CSS, reveal.js CSS) -> body (.reveal > .slides > sections) -> scripts (reveal.js init). Apply master layer configuration (logo, footer, slide numbers) per the theme.
      - Step 6: Apply animations per deck-plan.md recommendations. Use classes from animation-guide.md. Respect audience density rules.
      - Step 7: Apply German text handling: ensure `overflow-wrap: break-word` and `hyphens: auto` are in the global CSS.
      - Step 8: Write to `projects/{name}/presentation.html`.
      - Step 9: Verify by reading back the file and checking: correct number of sections, theme CSS linked, master layer present, all component classes applied.
    - `<success_criteria>`: presentation.html exists, has correct slide count matching deck-plan.md, theme CSS is linked, all reveal.js dependencies loaded, master layer configured, opens in browser without errors.
